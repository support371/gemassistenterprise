# 4.2 Threat Model (v0.1)

## Purpose
- Identify primary threats to GEM Cyber platform assets and define mitigation posture and residual risks.

## Scope
- Includes public intake, command center operations, IAM, data stores, and audit/event backbone.
- Excludes low-impact cosmetic/UI-only threats.

## Key decisions
- Threat model uses trust-boundary and abuse-case analysis aligned to STRIDE categories.
- Priority threats are those with compliance, integrity, or operational continuity impact.
- Sensitive action paths require auditable controls and deterministic ownership.

## Critical assets
- Identity and access state (users/roles/grants/sessions).
- Operational records (alerts/incidents/SLA/on-call state).
- Contact/subscriber PII.
- Audit events and compliance evidence.
- Publishing workflows (campaign schedule/dispatch state).

## Trust boundaries
- Internet -> Public web/API boundary.
- Public boundary -> Privileged command center boundary.
- API layer -> Domain data stores.
- Domain services -> Immutable audit store.
- Platform -> External integration endpoints.

## Threat scenarios and controls
- T1: Unauthorized admin API access
- Threat: bypass UI protections and call privileged APIs directly.
- Impact: high (operational integrity + compliance).
- Controls:
- Enforced auth checks on all privileged routes.
- Tenant/actor context validation.
- Audit logging for sensitive calls and failures.
- Residual risk: medium until stronger user/session IAM replaces shared token model.

- T2: Privilege escalation via mis-scoped grants
- Threat: role/grant misconfiguration yields excessive access.
- Impact: high.
- Controls:
- Explicit role-permission mapping and scope model.
- Immutable audit trail for grant changes.
- Governance review workflow.
- Residual risk: medium until policy-as-code and approval gates are enforced.

- T3: Tenant data leakage (cross-tenant read/write)
- Threat: missing tenant checks expose or modify another tenantâ€™s data.
- Impact: critical.
- Controls:
- Tenant scoping in all command center domains and APIs.
- Query-level tenancy filters and mutation checks.
- Audit traces tied to tenant context.
- Residual risk: medium; requires DB-enforced tenancy constraints in Phase 5.

- T4: Tampering with audit records
- Threat: delete/alter records to hide activity.
- Impact: critical (compliance/forensics failure).
- Controls:
- Append-only audit model.
- Restricted write path through audit backbone.
- Separate handling and retention class.
- Residual risk: medium until cryptographic signing/hash chains are added.

- T5: PII misuse from contact/newsletter channels
- Threat: exposure, over-retention, unauthorized use.
- Impact: high.
- Controls:
- Data classification and least-privilege access.
- Validation and structured intake handling.
- Audit visibility for sensitive mutations.
- Residual risk: medium pending formal retention/deletion automation.

- T6: Workflow abuse (incident/SLA manipulation)
- Threat: falsified status transitions or SLA resets.
- Impact: high.
- Controls:
- Status transitions via controlled APIs only.
- Event traces for state changes.
- Compliance review of breach outcomes.
- Residual risk: medium-low after separation-of-duty controls.

- T7: Supply-chain/integration compromise
- Threat: external connector abuse (SMTP now, SIEM/SSO later).
- Impact: medium-high.
- Controls:
- Principle of minimum external trust.
- Explicit integration boundaries and scoped credentials.
- Alerting on integration failures/anomalies.
- Residual risk: medium.

- T8: Denial of service on public intake
- Threat: volumetric or abusive submissions degrade operations.
- Impact: medium-high.
- Controls:
- Rate limiting and abuse protections.
- Operational visibility and fallback intake strategy.
- Residual risk: medium until formal edge protection policies are finalized.

## Highest-risk residuals (v0.1)
- Shared admin token model for privileged access.
- File-backed mutable stores instead of DB with strong constraints.
- Audit append-only without cryptographic integrity chain.

## Assumptions (explicit)
- Current baseline remains modular but not yet split into independently deployed services.
- Threat telemetry integration with external SIEM is target-state, not hard launch dependency.
- Insider threat and misconfiguration risks are in-scope and treated as primary.

## Non-blocking open questions
- Required adversary simulation cadence and ownership.
- Required evidence standard for non-repudiation (hash chain, signature, WORM).
- Whether break-glass access is required for launch and how approval is governed.
