# 4.4 Compliance Mapping (v0.1)

## Purpose
- Map architecture capabilities to compliance obligations and required evidence artifacts.

## Scope
- High-level control mapping for SOC2/ISO/NIST-aligned posture with operational and audit evidence alignment.
- Excludes formal auditor control narratives and policy text bodies.

## Key decisions
- Compliance mapping is capability-centric:
- IAM and least privilege.
- Data protection and classification.
- Change traceability and audit integrity.
- Operational resilience and response controls.
- Evidence must be generated from system behavior, not manual-only attestations.

## Compliance capability mapping
- Access Control & Identity Governance
- Architecture domains: IAM, Admin APIs, tenant context handling.
- Obligations:
- Enforce authentication/authorization for privileged actions.
- Apply least privilege and scope controls.
- Evidence:
- Role/grant event history, denied access events, privileged access logs.

- Auditability & Non-Repudiation
- Architecture domains: Audit/Event Backbone, command center mutation workflows.
- Obligations:
- Immutable records for sensitive operations.
- Trace actor and request attribution for key actions.
- Evidence:
- Append-only audit logs, export records, mutation-to-event linkage.

- Security Monitoring & Incident Handling
- Architecture domains: Threat Monitoring, Incident Response, SLA/On-call, Observability.
- Obligations:
- Detect, triage, and manage incidents with defined escalation paths.
- Track response commitments and breach handling.
- Evidence:
- Alert/incident timelines, SLA breach register, on-call action records.

- Data Protection & Privacy
- Architecture domains: Data classification, intake/newsletter stores, operational data boundaries.
- Obligations:
- Protect Confidential/Restricted data at rest/in transit.
- Limit access and enforce data handling constraints.
- Evidence:
- Classification model, access logs, retention/deletion records (target-state hardening).

- Change Management & Governance
- Architecture domains: Service/domain mutation paths, policy/grant changes.
- Obligations:
- Controlled and traceable changes to critical security and access configuration.
- Evidence:
- Grant/policy change events, release/change records, approval traces (target-state).

- Resilience & Availability
- Architecture domains: diagnostics, operational stores, continuity controls.
- Obligations:
- Maintain service continuity and recover critical operational/audit data.
- Evidence:
- Availability metrics, incident postmortems, backup/restore test records (target-state).

## Control-to-domain traceability (draft)
| Control Area | Primary Domain(s) | Evidence Source |
|---|---|---|
| Access Restriction | IAM, Admin API | Auth checks, denied access logs, role-grant events |
| Tenant Isolation | IAM, Data access layer | Tenant-scoped query/mutation logs, audit events |
| Incident Governance | Incident, SLA, On-call | Incident lifecycle events, breach records, escalation logs |
| Audit Integrity | Audit/Event Backbone | Append-only event store, audit query/export trails |
| Privacy Handling | Intake, Newsletter, Data architecture | Data classification docs, access logs, retention workflows |
| Monitoring Coverage | Observability strategy | Metrics/logs/traces and security alert definitions |

## Assumptions (explicit)
- Baseline aligns to enterprise compliance trajectory but requires additional operational controls for audit-readiness.
- Compliance evidence collection will increasingly be automated via event-driven telemetry.
- No immediate launch dependency on external SIEM/ticketing compliance integration is mandated.

## Non-blocking open questions
- Priority order of control families for first external audit cycle.
- Required customer-facing compliance reporting granularity by tenant tier.
- Whether contractual controls require per-tenant key segregation from launch.
