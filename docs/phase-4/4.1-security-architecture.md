# 4.1 Security Architecture (v0.1)

## Purpose
- Define the target security architecture for GEM Cyber across identity, data, application, and operational layers.

## Scope
- Covers platform-level controls for public site, authenticated command center, domain APIs, and data stores.
- Excludes vendor-specific implementation details and infrastructure-as-code.

## Key decisions
- Zero-trust posture for all authenticated surfaces.
- Tenant-scoped authorization is mandatory for all command center operations.
- Defense-in-depth model with layered controls:
- Identity and access controls.
- Application/API protections.
- Data protection controls.
- Monitoring/detection controls.
- Response and recovery controls.
- Audit backbone remains append-only and is treated as high-integrity evidence data.

## Security architecture layers
- Identity & Access Layer
- Strong authentication for privileged users.
- Role- and permission-based authorization with tenant scope.
- Session controls (expiry, rotation, revocation) and privileged action checks.

- Application/API Layer
- Centralized authz checks for sensitive APIs.
- Input validation, payload constraints, and request-context tracing.
- Abuse controls (rate limiting, endpoint hardening, anti-automation controls).

- Data Protection Layer
- Classification-aware handling (Public/Internal/Confidential/Restricted).
- Encryption in transit and at rest for Confidential/Restricted data.
- Controlled export pathways for audit/compliance data.

- Event & Audit Layer
- Append-only audit/event records.
- Immutable evidence posture for sensitive operations.
- Traceability from actor/request to state mutation.

- Detection & Response Layer
- Operational telemetry and alerting for auth failures, privilege changes, and abnormal workflow patterns.
- Incident response hooks tied to SLA and escalation domains.

- Continuity & Resilience Layer
- Backups, restore procedures, and retention controls aligned to data class.
- Recovery objectives defined by operational criticality (incident/audit first).

## Security boundaries
- Public boundary: unauthenticated web and contact/newsletter intake.
- Privileged boundary: `/admin/*` UX and `/api/admin/*` APIs.
- Data boundary: tenant-scoped operational data and restricted audit/evidence stores.
- Integration boundary: SMTP and future SSO/SIEM/ticketing connectors.

## Control objectives (high-level)
- Prevent unauthorized access and lateral privilege misuse.
- Ensure integrity and non-repudiation of critical operational and audit records.
- Minimize blast radius through tenancy isolation and role scoping.
- Detect and respond rapidly to security events and control failures.

## Assumptions (explicit)
- Multi-tenant model remains in scope.
- GEM operates the platform and core security controls.
- No mandatory customer-managed key requirement from day one.
- No hard EU-only residency launch constraint has been declared.

## Non-blocking open questions
- Required MFA strength and policy for each privileged role class.
- Target recovery metrics (RTO/RPO) for audit and incident datasets.
- Formal key rotation cadence and separation-of-duty requirements.
