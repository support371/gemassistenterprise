# 4.3 Audit and Observability Strategy (v0.1)

## Purpose
- Define how GEM Cyber captures auditable evidence and operational telemetry to support security response, compliance, and reliability.

## Scope
- Covers logs, metrics, traces, domain events, and audit records across public and privileged workflows.
- Excludes specific vendor tooling selections.

## Key decisions
- Distinguish operational telemetry from compliance-grade audit evidence.
- Treat audit events as immutable and retention-governed.
- Require request correlation (`requestId`) and actor attribution (`actorId`) for sensitive operations.
- Monitoring strategy is domain-aware (IAM, incident, SLA, on-call, newsroom).

## Observability pillars
- Logs
- Structured application and security logs for API access, auth outcomes, and workflow mutations.
- Sensitive fields redacted by default.

- Metrics
- Availability/latency/error metrics for APIs and command center workflows.
- Security posture metrics (auth failures, denied access, unusual mutation rates).
- Domain metrics (open incidents, SLA breaches, escalation latency, campaign queue depth).

- Traces
- Request-level tracing across API entry -> domain mutation -> audit emission.
- Required for privileged operations and high-impact workflows.

- Audit Events
- Append-only immutable records for sensitive changes.
- Queryable by tenant, event type, entity, time.
- Export pathway with governance controls.

## Audit event minimum schema (logical)
- `eventId`
- `timestamp`
- `tenantId`
- `actorId`
- `requestId`
- `eventName`
- `entityType`
- `entityId`
- `payload` (sanitized, structured, classification-aware)

## Event classes
- Security events: auth failures, unauthorized attempts, grant changes.
- Operational events: incident/alert status changes, SLA breach records, on-call actions.
- Governance events: policy changes, evidence updates, export actions.
- Publishing events: subscription/campaign scheduling/dispatch outcomes.

## Detection and alerting strategy
- High severity alerts:
- Repeated unauthorized admin API attempts.
- Cross-tenant access anomalies.
- Audit write failures.
- Incident/SLA anomalies (clock regressions, status tampering patterns).

- Medium severity alerts:
- Elevated failed logins, unusual campaign scheduling bursts, unexpected diagnostics failures.

- Escalation path:
- Alert -> on-call route -> incident creation if threshold/pattern confirms risk.

## Retention and access posture
- Operational logs: medium retention, role-limited access.
- Audit events: long retention, immutable handling, restricted export.
- Compliance evidence: controlled retention with legal-hold capability.
- Access to Restricted-class observability data requires explicit privilege.

## Governance model
- Security engineering owns detection coverage and control telemetry.
- Compliance owns evidence requirements and audit export governance.
- Platform operations own reliability telemetry and incident coordination.

## Assumptions (explicit)
- Existing append-only audit approach is accepted as v0.1 baseline.
- Full SIEM pipeline can be staged after baseline telemetry normalization.
- Domain-level event taxonomy is stable enough to support phased hardening.

## Non-blocking open questions
- Required MTTD/MTTR reporting granularity for governance review.
- Expected retention periods by jurisdiction and tenant contract class.
- Whether customer-facing audit exports are required in launch scope.
