# 3.3 Service Responsibility Matrix (v0.1)

## Purpose
- Define service-level responsibility boundaries, core inputs/outputs, dependencies, and control obligations.

## Scope
- Draft matrix for logical services/modules aligned with bounded contexts.
- Includes platform services and cross-cutting governance responsibilities.

## Key decisions
- Every service has explicit data classification and audit obligations.
- Tenant scope is mandatory for all operational service contracts.
- Cross-service event naming and versioning are standardized.

## Matrix
| Service / Module | Primary Responsibility | Inputs | Outputs | Depends On | Control Obligations |
|---|---|---|---|---|---|
| API Gateway / Web API Layer | Unified HTTP entry, auth gate, request shaping | Browser/API requests | API responses, request context | IAM, Domain modules | AuthN/AuthZ enforcement, request IDs, rate controls |
| IAM Service | Identity, roles, permissions, tenant scoping | Login/session info, policy changes | Access decisions, grants, session state | Audit backbone | Least privilege, tenant isolation, auditable grant changes |
| Contact Intake Service | Contact capture and triage workflow entry | Contact form submissions | Contact records, triage updates | IAM, Audit, storage | PII protection, validation, mutation audit trail |
| Threat Monitoring Service | Normalize and manage alert pipeline | Detection events, intel signals | Alerts, escalation triggers | IAM, Incident service, Audit | Severity governance, traceable status changes |
| Incident Response Service | Incident lifecycle and operational coordination | Escalated alerts, analyst actions | Incident records, status transitions | SLA, On-call, Audit | Immutable timeline semantics, post-incident traceability |
| SLA Service | Clock tracking and breach detection | Incident state/timestamps, SLA profiles | Breach events, SLA status | Incident service, Audit | Time integrity, breach non-repudiation |
| On-Call Service | Coverage schedule and escalation control | Alerts/incidents/SLA warnings | Paging/escalation outcomes | IAM, Incident/SLA, Audit | Rotation integrity, ack/action logging |
| Newsroom Service | Subscriber/campaign operations and publishing flow | Subscription events, campaign requests | Campaign queue, dispatch records | IAM, Audit, storage | Subscription compliance, dispatch logging |
| Compliance & Governance Service | Controls/evidence posture and report readiness | Incidents, breaches, policy updates | Control status, evidence bundles | Audit, IAM, Incident/SLA | Evidence integrity, export accountability |
| Audit & Event Backbone | Append-only event recording and query | Domain events from all services | Immutable audit ledger, filtered event views | Secure storage | Tamper evidence, retention controls, restricted access |
| Notification Service (target) | Multi-channel operational notifications | Alert/incident/SLA/news events | Sent/failed delivery records | Incident/SLA/Newsroom, Audit | Delivery traceability, escalation guarantees |
| Data Access Layer / Repositories | Persistence abstraction for domain modules | Domain state mutations/queries | Persisted domain data | DB and storage systems | Schema validation, tenancy constraints, migration safety |

## Assumptions (explicit)
- Service boundaries are logical modules first, then physically separable services.
- Notification and worker orchestration are target-state modules, not day-one hard dependencies.
- Compliance service consumes domain events rather than directly mutating operational records.

## Non-blocking open questions
- Ownership split between SOC operations and compliance engineering for SLA breach adjudication.
- Whether newsroom remains internal-only or adds tenant self-service controls.
- Required SLA for audit query latency under incident surge conditions.
