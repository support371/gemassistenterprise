# 3.1 C4 Context Diagram (Text) (v0.1)

## Purpose
- Describe the system context boundaries, external actors, and upstream/downstream dependencies in C4 Level 1 textual form.

## Scope
- System context only (no internal service breakdown).
- Focus on GEM Cyber enterprise platform interactions.

## Key decisions
- GEM Cyber Platform is the central system for public engagement, command center operations, and governance evidence.
- External integrations are modeled as optional unless immediate hard dependency is declared.
- Human actors are separated by operational and governance responsibilities.

## System context (textual)
- Central System:
- `GEM Cyber Enterprise Platform`
- Capabilities: public website, command center, operational security workflows, IAM, audit backbone, newsroom operations.

- Primary Actors:
- `Public Visitor`: consumes public content, submits contact requests, subscribes to newsletter.
- `SOC Analyst`: monitors alerts, triages incidents, manages intake handoff.
- `Incident Commander`: drives incident lifecycle, coordinates containment/resolution.
- `Compliance Lead`: reviews SLA breaches, audit events, and evidence outputs.
- `Platform Admin`: manages IAM, tenant scope administration, diagnostics and governance controls.
- `Newsroom Operator`: schedules campaigns and manages subscriber/publishing workflows.

- External Systems (current/target-adjacent):
- `SMTP Provider` (current optional): outbound contact acknowledgement and notifications.
- `Threat/Telemetry Sources` (target): upstream detections feeding alert normalization.
- `Identity Provider (SSO)` (target): enterprise auth assertions and identity lifecycle federation.
- `SIEM/Ticketing Platforms` (target): bidirectional enrichment and workflow synchronization.
- `Compliance/Audit Consumers` (target): exports and evidence package recipients.

- Data/Interaction Flows:
- Public user -> platform: contact/newsletter submissions.
- Platform -> SOC/IR users: alerts/incidents/SLA state surfaces.
- Platform -> compliance users: audit and governance reports.
- Platform -> external comms systems: notification dispatch.
- Platform <-> external security tooling: event ingestion and contextual updates (planned).

## Assumptions (explicit)
- Platform is GEM-operated and centrally managed.
- Multi-tenant boundary exists inside platform scope.
- External integrations beyond SMTP are not hard launch blockers in v0.1.

## Non-blocking open questions
- Priority order for first mandatory external integration after core hardening.
- Whether customer-facing tenant admins are introduced in v0.2 scope.
