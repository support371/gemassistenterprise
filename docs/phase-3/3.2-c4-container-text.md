# 3.2 C4 Container Diagram (Text) (v0.1)

## Purpose
- Describe runtime/application containers and their responsibilities in C4 Level 2 textual form.

## Scope
- Logical containers and major interactions.
- No deployment topology or infra-specific sizing.

## Key decisions
- Keep public and command center UX in one web application container initially.
- Separate domain data stores by concern even if physically co-located during early phases.
- Treat audit/event store as append-only and logically independent from mutable operational stores.
- Use API-first interfaces for all command center domain operations.

## Containers (textual)
- `Web App Container (Next.js)`
- Responsibilities: public pages, authenticated command center UI, route handling, API endpoints.
- Interfaces: browser HTTP(S), internal domain store adapters.

- `IAM Policy/Session Module`
- Responsibilities: auth checks, role/permission enforcement, tenant-scoped context extraction.
- Interfaces: consumed by admin APIs and protected pages.

- `Operations Domain Module`
- Responsibilities: alerts, incidents, SLA profiles/breaches, on-call schedules.
- Interfaces: `/api/admin/alerts`, `/api/admin/incidents`, `/api/admin/sla`, `/api/admin/on-call`.

- `Intake Domain Module`
- Responsibilities: contact message ingestion/triage, subscriber lifecycle.
- Interfaces: `/api/contact`, `/api/admin/inbox`, `/api/newsletter`.

- `Newsroom Domain Module`
- Responsibilities: campaign scheduling and intelligence publication support workflows.
- Interfaces: `/api/admin/newsroom/campaigns`.

- `Audit/Event Backbone Module`
- Responsibilities: append-only event recording and query/filter for governance.
- Interfaces: internal event emitters and `/api/admin/audit/events`.

- `Mutable Operational Store` (current file-backed, target DB-backed)
- Responsibilities: mutable state for inbox, command center entities, IAM snapshots, newsletters.

- `Immutable Audit Store` (append-only)
- Responsibilities: audit event retention for forensics/compliance.

## Container interactions
- Browser -> Web App: page rendering + authenticated actions.
- Web App APIs -> IAM module: request authentication and tenant scoping.
- Web App APIs -> Domain modules: perform business operations.
- Domain modules -> Audit/Event module: emit auditable domain events.
- Domain modules -> Mutable store: persist operational state.
- Audit/Event module -> Immutable store: append-only writes.

## Assumptions (explicit)
- Single web app container is acceptable for v0.1 documentation baseline.
- Modularization boundaries are stable enough to split into independent services later.
- Queue/worker layer can be introduced after DB hardening without route contract breaks.

## Non-blocking open questions
- First container to externalize into a separate service boundary (likely Incident/SLA).
- Whether audit query should remain in-app or move to dedicated compliance service.
